# Architecture and Technology Stack

This document provides a technical overview of the DisherIo platform architecture and its underlying technology stack.

---

## System Architecture

The platform utilizes a monorepo architecture organized into three primary modules:
- **backend**: Node.js Express API facilitating business logic, data persistence, and real-time communication.
- **frontend**: Angular SPA providing the user interface across various modules.
- **shared**: Common library containing unified types, Zod schemas, and error code definitions.

### Service topology

```
                    Internet
                       │
               ┌───────┴───────┐
               │  Caddy :80/:443 │  ← TLS termination and reverse proxy
               └───────┬───────┘
                      │
         ┌────────────┼────────────┐
         │            │            │
    Frontend       Backend      /uploads/*
    Angular 21    Express 5    (Caddy serves
    Caddy :4200   Node :3000    from shared volume)
         │            │
         │     ┌──────┴──────┐
         │     │             │
         │  MongoDB       Redis
         │  :27017        :6379
         │  (rs0)         (cache + Socket.IO
         │                 adapter + token
         │                 blocklist)
```

### Request routing

| Path | Routed to | Purpose |
|------|-----------|---------|
| `/api/*` | `backend:3000` | REST API (auth, dishes, orders, payments, staff, totems) |
| `/socket.io/*` | `backend:3000` | WebSocket (Socket.IO with `flush_interval -1`) |
| `/uploads/*` | Caddy file_server | Static images from `disherio_uploads` volume (`/srv/uploads`) |
| `/health/*` | `backend:3000` | Health endpoints (`/health`, `/health/ready`, `/health/live`, `/health/simple`) |
| `/metrics` | `backend:3000` | Internal Prometheus exposition generated by `prom-client`; not routed by Caddy |
| `/*` | `frontend:4200` | Angular SPA (Caddy `try_files` → `index.html`) |

MongoDB, Redis, and one-shot initialization or seed services. Grafana,

## Architectural Design Patterns

### Repository Pattern (Backend)

The backend implements the Repository Pattern to decouple business logic from the data access layer. Services interact exclusively with Repositories, which manage Mongoose model interactions.

Key repository: `BaseRepository` provides `validateObjectId()` and `new Types.ObjectId()` for all queries — preventing NoSQL injection.

### Service Layer Pattern

Controllers are thin: they extract validated input and call services. Services contain business logic and use repositories for data access. Error handling via `asyncHandler` wrapper.

### Authorization Model (CASL)

Abilities are defined in `backend/src/abilities/abilities.ts`:

| Role | Permissions |
|------|-------------|
| Admin | `manage` all subjects |
| POS | read/create/update Orders, ItemOrders, Payments, Totems |
| TAS | read/create Orders, ItemOrders, Payments, Customers; manage temporary totems |
| KTS | read Orders/ItemOrders/KDS, update ItemOrders, update Dish availability only |

Middleware: `requirePermission(action, subject)` on every route. Controllers also verify `restaurant_id` ownership (anti-IDOR).

### Authentication Model

- **Access token**: JWT (15m), stored only in HttpOnly cookie `auth_token`
- **Refresh token**: Opaque 32-byte random token (7d), stored in Redis keyed by `refresh:<staffId>:<sha256(token)>`; rotation re-hydrates the payload from the database
- **Rotation**: refresh tokens are single-use; an atomic Redis rotation stores a consumed-token tombstone before reuse detection can revoke the entire family
- **Blocklist and authorization version**: access tokens are blocklisted in Redis on logout (TTL = remaining token lifetime). Every authenticated HTTP request and Socket.IO handshake also checks the token's `authVersion` against the current staff record. Credential or role assignment changes increment that version, so existing access tokens stop working immediately. Revocation checks fail closed if Redis or MongoDB is unavailable.
- **Audit**: refresh-token lifecycle events (issue, rotate, reuse, revoke, logout) are emitted as structured Pino logs and consumed from the log stream; they are not retained in process memory.
- **Cookies**: `httpOnly: true`, `secure: true` (HTTPS), `sameSite: strict`
- **PIN login**: staff authenticate with a 4-digit PIN verified by bcrypt. A deterministic `pin_lookup` key (HMAC-SHA256 of the PIN with `PIN_LOOKUP_PEPPER`, uniquely indexed per restaurant) narrows the candidate so login costs one indexed query plus a single `bcrypt.compare` regardless of staff count; without a candidate, a dummy compare equalizes timing. Documents created before the key existed are migrated opportunistically on their next successful PIN login. Duplicate PINs inside one restaurant are rejected.
- **Client renewal**: concurrent 401 responses share one refresh request; successful rotation retries the original requests
- **Socket revocation**: authenticated sockets join a staff-specific room. Logout, staff deletion, and security-relevant staff or role changes revoke refresh credentials and disconnect every active socket for that staff identity.

### Rate Limiting

| Limiter | Scope | Limit |
|---------|-------|-------|
| `authLimiter` | `/api/auth/*` | 5 failed attempts / 15 min |
| `apiLimiter` | `/api/*` | 1000 requests / 15 min |
| `strictLimiter` | Critical operations | 20 requests / 15 min |
| `uploadLimiter` | `/api/uploads/*` | 10 uploads / hour |
| `qrLimiter` | Public QR endpoints | 30 requests / min |
| `qrBruteForceLimiter` | QR token validation | 10 attempts / 15 min |

Key generator uses `userId` if authenticated, otherwise IP (via `ipKeyGenerator` with `trust proxy`).

### Public Totem Session Isolation

The printed QR token on a totem is a permanent bearer credential for a physical table. Possession of that QR bootstraps public access; the application cannot independently prove physical proximity. Every `TotemSession` also carries an ephemeral `session_token` (a UUID generated at session creation). The token binds later operations to the current session and invalidates credentials retained from a closed/reopened session. It is returned by `POST /totems/menu/:qr/session`, kept in per-tab session storage (never in the URL), and echoed back by the client:

- HTTP: in the `x-session-token` header for GET endpoints and in the `session_token` body field for POST endpoints.
- Socket.IO: in the `sessionToken` field of the `totem:join_session` payload.

The server validates the token on every public call and binds it to the socket at join time, re-validating against the database on subsequent `totem:*` events. Authenticated staff sockets are rejected from totem events.

A scan joins the current `STARTED` session for that table. When no active
session exists, it creates a new session regardless of historical closed
sessions, so a permanent table QR remains reusable for later visits. Customer
identity used by subsequent socket events comes only from the verified join
binding and cannot be overridden by event payloads.

Network placement changes the protection provided by this model. When the
application entry point is reachable only from a trusted restaurant LAN, the
network boundary prevents a person outside that LAN from reusing a photographed
QR. This property depends on firewall rules, the absence of public forwarding
or tunnels, and control of the restaurant Wi-Fi; it does not protect against a
device already connected to that network. When the application is reachable
from the public Internet (including a cloud VM, public domain, Cloudflare
Tunnel, or ngrok), a static QR and the derived `session_token` do not prove
physical presence. A remote holder of a photographed QR can bootstrap public
access. TLS, token rotation, and rate limits protect transport, session replay,
and automation respectively, but do not add proof of proximity.

An Internet-facing deployment that needs proximity enforcement must add an
independent presence gate, such as a session-specific code supplied at the
table, staff approval in POS/TAS, or a trusted on-premises network assertion.
DisherIo does not currently implement that gate; public deployments must treat
remote QR reuse as an accepted risk until one is configured or implemented.

Public Socket.IO connections must present the totem QR token in the handshake `auth` payload (`{ publicTotem: true, qr: '<token>' }`) and the server validates it against the database before accepting the connection; a bare `publicTotem: true` flag without a valid QR is rejected. Session state transitions (`STARTED → COMPLETE → PAID`, `STARTED → CANCELLED`, `COMPLETE → STARTED`) are atomic `findOneAndUpdate` operations filtered by the current state. Closing preserves the `COMPLETE` session in active POS/TAS views for payment; reopening rotates its session token in the same transactional write. Archiving creates a full-payment record when necessary and marks every ticket paid in the same transaction as `COMPLETE → PAID`, then removes the session from active views. Payments retain restaurant and table snapshots so history survives temporary-totem cleanup. `BY_USER` tickets use the same active-item snapshot as the payment total and allocate tip and rounding differences proportionally so ticket cents reconcile exactly with `payment_total`. The last ticket paid follows the same archive path and cleanup semantics. POS, TAS, and KDS sockets subscribe to restaurant-scoped rooms for discovery, while session rooms carry scoped updates. Close, cancel, reopen, and archive transitions update every connected session list without requiring the affected session to be selected. Close/cancel events carry the persisted target state, preventing a cancelled session from being rendered as pending payment. A partial unique MongoDB index allows at most one `STARTED` session per totem, while a unique payment index permits at most one `Payment` per session. Item mutations and public order-limit validation serialize on the same session document and reject any session with a payment snapshot. A session that already has a `Payment` record cannot be reopened.

### Internationalization

Backend uses `i18next` with file-system backend loading translations from `/app/dist/locales/{es,en,fr}/translation.json`. The `languageMiddleware` detects language from `Accept-Language` header. i18n initialization is non-blocking (catch fallback) so the backend starts even if a locale file is missing.

### Secret Redaction in Logs

Pino logger redacts the following fields from all log output:
- `req.headers.authorization`, `req.headers.cookie`
- `password`, `password_hash`, `pin`, `pin_code`, `pin_code_hash`, `pin_lookup`
- `token`, `access_token`, `refresh_token`
- `JWT_SECRET`, `JWT_REFRESH_SECRET`, `MONGO_ROOT_PASS`, `MONGO_APP_PASS`, `REDIS_PASSWORD`
- `MONGODB_URI`

HTTP request logs and Prometheus labels use Express route templates such as
`/api/totems/qr/:qr`; raw URLs, query values, QR tokens, and unmatched concrete
paths are not recorded.

### Security Headers (Caddy)

The domain-mode Caddyfile sets:
- `Strict-Transport-Security` (HSTS, 1 year, preload)
- `Content-Security-Policy` (restrictive: `default-src 'self'`)
- `X-Frame-Options: SAMEORIGIN`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy` (disables camera, mic, geolocation, etc.)
